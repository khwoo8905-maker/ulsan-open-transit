# Ulsan Open Transit (울산 버스 공개 진단)

울산 시내버스를 **공개데이터(data.go.kr)** 와 **AI 도구**만으로 진단하는 오픈 프로젝트입니다.
노선 구조 · 정류장별 승하차 · 대중교통 접근성을 모아 "수요와 접근성이 어긋난 곳"을 찾고, 개선 시나리오를 정량 비교합니다.

**by Dambaek (담백)** — 울산 기반 1인 개발자. 공개데이터와 AI 도구로 혼자 만듭니다.
> 공개데이터로 우리 동네 문제를 담백하게 들여다봅니다.

> **📌 읽기 전에 (이 자료의 성격)**
> - 저희는 **"이렇게 바꾸라"는 해법을 내놓지 않습니다.** 누구나 같은 코드로 직접 돌려볼 수 있는 **도구와 방법**을 공개할 뿐입니다.
> - 특정 노선개편(2024-12-21)에 대한 **찬반 자료가 아닙니다.** 누구나 재현할 수 있는 중립적 **진단 도구**입니다.

## 핵심 발견 (요약)
울산은 **다핵도시**라 "시청 하나"로 접근성을 재면 그림이 왜곡됩니다. 그래서 생활권 **거점 8개**(시청·삼산상권·태화강역·울산대·현대중공업·울산대병원·KTX울산역·언양)까지의 접근성을 측정했습니다. (평일 08시, 수요 = 실제 승하차)

- **시민 9.4%(하루 약 2.8만 명)는 8개 주요 거점 어디에도 45분 내 도달 못 함.** 45분 내 닿는 거점은 평균 3.0개뿐.
- **거점별 격차가 큼**: 도심(시청·삼산 37~40분)은 양호하나 **현대중공업(동부 고용) 60분 · 울산대병원(의료) 66분 · KTX울산역 76분 · 언양 81분**.
- **측정 기준을 '시청 한 곳'으로 좁히면 동구가 최악으로 보이지만, 거점 8개로 넓히면 그림이 달라짐** — 동구엔 동네 일자리(현대중공업)가 있어 시청 통행 수요는 적고, 대신 **도심·동부 의료를 잇는 환승이 실제 병목**입니다. 거점 전체로 본 단절 후보는 ① 남구 도심 ↔ 동부 산업·의료 ② 전 도시 ↔ 광역 서부(KTX·언양) ③ 의료 접근성 도시 전반.
- 노선을 되돌리거나 새로 그리는 것만으로는 부족하고 **수요-접근성**까지 함께 봐야 격차가 풀립니다 — '우리 노선이 유난히 비효율인가'까지 같은 도구로 점검할 수 있습니다(방법은 저장소에).
- **(실행 규모)** 정류장 **1,498곳 × 거점 8개 ≈ 1.2만 개 경로**를 평일 08시로 계산하고, 노선 **191개**의 첫·막차까지 훑었습니다(좌표 조인 정류장 93%·승차량 91%). 코드·데이터를 전부 공개해 **누구나 그대로 다시 돌릴 수 있습니다.** *(단일 평일·모델 배차 기준이라는 한계는 그대로 → [LIMITATIONS.md](LIMITATIONS.md))*

> 이 프로젝트는 **정책 처방이 아니라 재현 가능한 진단 도구**입니다. 절대 소요시간은 근사치이며 **상대 순위·패턴**이 메시지입니다. → [LIMITATIONS.md](LIMITATIONS.md)

## 저장소 구조
```
config/ulsan.yaml     분석 설정 (거점·입력·기준) — 도시 교체 시 이 파일만 교체
pull_routes.py        노선·정류장 수집 (TAGO)
pull_city.py          타 도시 수집 (cityCode 인자 → 벤치마크)
collect_demand.py     정류장별 승하차 수집 (브라우저 UA 필요)
collect_realtime.py   실시간 버스 위치 수집 (배차·정시성 실측용)
build_gtfs.py         수집 데이터 → GTFS
accessibility.py      ★ 다목적지 접근성 진단 (config 기반, r5py)   [Linux/JDK21]
diagnose_access.py    초기 단일거점 진단 (r5py)                     [Linux/JDK21]
make_access_map.py    접근성 지도 (folium, config 기반)
build_dataset.py      공개 데이터셋 패키징
dataset/              공개 CSV + 데이터셋 카드
experiments/          보조 처방 시뮬(급행 신설 등) — 본편 아님
REPORT.md             방법·발견 리포트
METHODOLOGY.md        분석 파라미터(45분·8거점·08시 등)의 근거
LIMITATIONS.md        한계와 주의 (필독)
requirements.txt      재현용 의존성 (r5py는 OpenJDK 21 필요)
```

## 재현 방법
1. `cp .env.example .env` 후 data.go.kr 인증키 입력
2. `python pull_routes.py` → `python collect_demand.py` → `python build_gtfs.py`
3. 접근성 진단은 `r5py`(OpenJDK 21) 환경에서 `python accessibility.py -c config/ulsan.yaml`
4. 지도는 `python make_access_map.py`, 데이터셋은 `python build_dataset.py`

**다른 도시 적용**: 수집은 `cityCode`만, 분석은 `config/ulsan.yaml`(거점·입력·기준)만 바꾸면 동일하게 재실행됩니다.
