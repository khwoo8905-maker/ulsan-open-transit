# 방법론 (METHODOLOGY)

> 이 문서는 분석에 들어간 **선택과 가정의 근거**를 밝힙니다.
> "왜 45분인가", "왜 거점 8개인가" 같은 수치가 **자의적이지 않음**을 보이기 위한 것이며,
> 모든 값은 [`config/ulsan.yaml`](config/ulsan.yaml)에서 바꿀 수 있어 누구나 다른 가정으로 재실행할 수 있습니다.
> 한계(무엇을 못 하는지)는 별도 [LIMITATIONS.md](LIMITATIONS.md)를 함께 보세요.

---

## 0. 한눈에 보는 분석 파라미터

| 항목 | 값 | 근거 요약 |
|------|----|-----------|
| 출발 시각 | 평일 08:00 | 출근 피크 — 대중교통 공급이 가장 촘촘하고 통근 수요가 집중되는 시간 |
| 이동 수단 | 대중교통 + 도보 | 버스 중심 도시의 실제 통근 조합 (자가용 제외 = 대중교통 의존층 관점) |
| 접근 등급 구간 | 15 / 30 / 45 / 60분 | 통근 감내 시간의 관례적 단계 |
| "도달 가능" 기준 | 45분 이내 | 통근 1회 편도 감내 한계의 통용 기준 |
| 생활권 거점 | 8개 | 다핵도시 울산의 대표 통행 목적지 표본 |
| 라우팅 엔진 | r5py (R5) | 표준 GTFS+OSM 기반 시간대 의존 대중교통 경로계산 오픈소스 |

---

## 1. 왜 "평일 오전 8시 출발"인가
- 대중교통 접근성은 **출발 시각에 따라 크게 달라집니다**(배차·막차). 단일 기준이 필요해 **통근 피크**를 골랐습니다.
- 08:00은 출근 통행이 집중되고 버스 공급(배차)이 가장 조밀한 시간대로, **"가장 유리한 조건에서도 얼마나 걸리는가"**를 보는 보수적 기준이 됩니다. 심야·주말은 이보다 나쁩니다.
- 다른 시각으로 보려면 `config/ulsan.yaml`의 `analysis.departure`만 바꾸면 됩니다.

## 2. 왜 "45분"을 도달 기준으로 쓰는가
- 45분은 **편도 통근 감내의 관례적 상한**으로 널리 쓰이는 구간입니다(국내외 통근통행 분석에서 30·45·60분 단계가 통용).
- 절대적 정답이 아니라 **비교 기준선**입니다. 그래서 15/30/45/60분을 모두 계산해 두고, "거점 N개 도달" 집계에는 45분을 씁니다.
- 30분이나 60분으로 바꾸면 비율이 달라집니다 → `config`의 `within_min`으로 조정해 재현 가능. **수치 자체보다 지역 간 상대 순위가 메시지입니다.**

## 3. 왜 "거점 8개"이고, 어떻게 골랐나
- 울산은 시청 하나로 통행이 수렴하지 않는 **다핵도시**입니다. 단일 거점(시청)으로 접근성을 재면 동부 산업지대처럼 "동네에 일자리가 있어 시청에 갈 일이 적은" 지역을 왜곡합니다.
- 그래서 **기능이 다른 8개 거점**을 골라 다목적지로 측정했습니다:

| 거점 | 기능 범주 |
|------|-----------|
| 시청 | 도심·행정 |
| 삼산·현대백화점 | 도심·상업 |
| 태화강역 | 철도환승 |
| 울산대학교 | 교육 |
| 현대중공업 | 동부·고용 |
| 울산대학교병원 | 동부·의료 |
| KTX울산역 | 광역·서부 |
| 언양터미널 | 광역·서부 |

- 이는 **전체 목적지가 아니라 대표 표본**입니다(행정·상업·고용·의료·교육·철도·광역을 포괄하도록 선정). 좌표는 GTFS/승하차 데이터의 실제 정류장에서 확정했습니다.
- 거점 추가·교체는 `config/ulsan.yaml`의 `destinations`에서 가능합니다.

## 4. 배차·운행속도 가정 (자체 가공 GTFS)
- 울산시는 표준 GTFS를 공개하지 않아, 공개 노선·정류장 데이터로 GTFS를 **직접 생성**했습니다(`build_gtfs.py`).
- 배차 간격·평균 운행속도는 **노선 종류별(간선/지선/외곽) 표준값으로 가정**했습니다.
- → 따라서 산출되는 **절대 분(分)은 근사값**이며, 의미 있는 것은 **정류장·지역 간 상대 격차**입니다. (상세 한계 → LIMITATIONS §1)
- 가공 로직 전체가 코드로 공개돼 있어 가정을 바꿔 재계산할 수 있습니다.

## 5. 수요 가중치 = 실제 승하차 실적
- 접근성 불리 지역의 **체감 영향**을 보려면 인구가 아니라 실제 이용을 봐야 합니다. 그래서 **국가대중교통DB의 정류장별 승하차 인원**을 수요 가중치로 씁니다.
- 단, 승하차는 *이미 버스가 다녀서* 생긴 수요라, **공급이 없어 억압된 잠재수요는 포착되지 않습니다**(LIMITATIONS §5). "저수요"를 "수요 없음"으로 읽지 않도록 주의합니다.

## 6. 좌표 조인
- 승하차 데이터에 좌표가 없어 **정류장명 정규화·접두 매칭**으로 좌표를 붙였습니다.
- 커버리지: 정류장 약 93% / 승차량 약 91%. 이름이 크게 다른 일부는 누락됩니다(LIMITATIONS §6).

## 7. 라우팅 — r5py (R5)
- 경로계산은 오픈소스 [r5py](https://r5py.readthedocs.io)(Conveyal R5 래퍼)를 사용합니다. GTFS+OSM 도로망 위에서 **시간대 의존 대중교통+도보** 경로를 계산합니다.
- **이상적 정시 운행**을 가정합니다(지연·혼잡·결행 미반영, LIMITATIONS §2). 실시간 정시성은 `collect_realtime.py`로 별도 수집해 추후 보정 예정입니다.
- 실행에는 **OpenJDK 21+**가 필요합니다(requirements.txt 참고).

## 8. 검증 (권장)
- 절대 수치는 **네이버/카카오 길찾기 등 현장 자료와 대조**해야 합니다.
- 이 도구의 역할은 "어디가 상대적으로 불리한가"를 빠르게 드러내는 것이고, 최종 판단은 현장과 함께 사람이 내립니다.

## 재현 순서 요약
```
cp .env.example .env          # data.go.kr 인증키 입력
python pull_routes.py         # 노선·정류장 수집 (TAGO)
python collect_demand.py      # 정류장별 승하차 수집
python build_gtfs.py          # 수집본 → GTFS
python accessibility.py -c config/ulsan.yaml   # 다목적지 접근성 (OpenJDK 21)
python make_access_map.py     # 접근성 지도
python build_dataset.py       # 공개 데이터셋 패키징
```
모든 파라미터는 `config/ulsan.yaml` 한 곳에서 조정됩니다 — 다른 가정·다른 도시로 그대로 재실행 가능합니다.
